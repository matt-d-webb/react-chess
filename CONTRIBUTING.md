# Contributing to react-chess

Thanks for your interest in contributing! This document explains how the repo is laid out, how to work on it locally, and how releases happen.

## Repo layout

This is a [Turborepo](https://turbo.build/) + [pnpm workspaces](https://pnpm.io/workspaces) monorepo:

```
react-chess/
├── packages/
│   └── react-chess/        # The library — published to npm as @mdwebb/react-chess
│       ├── src/            # Library source (components, hooks, themes, types)
│       ├── dist/           # Build output (gitignored, generated by rollup)
│       ├── rollup.config.js
│       └── package.json
├── apps/
│   └── web/                # Next.js demo site (consumes the lib via workspace link)
├── docs/                   # Static assets (screenshots, etc.)
├── turbo.json
├── pnpm-workspace.yaml
└── package.json            # Monorepo root (private, no version field)
```

The demo at `apps/web` imports `@mdwebb/react-chess` directly through the pnpm workspace, so any change to `packages/react-chess/src` is picked up in the demo without re-publishing.

## Prerequisites

- **Node.js** 24 LTS (or any active LTS — `nvm use` if you have an `.nvmrc`)
- **pnpm** 10.x (`npm i -g pnpm` if you don't have it)

## Getting started

```sh
pnpm install
pnpm dev
```

`pnpm dev` runs both targets in parallel via turbo:

- `packages/react-chess` — rollup in watch mode, emitting to `dist/`
- `apps/web` — Next.js dev server (typically http://localhost:3000)

Editing files in `packages/react-chess/src` triggers a rebuild, and Next.js HMR picks up the new `dist/` output.

## Useful scripts

Run from the repo root unless noted otherwise.

| Command | What it does |
|---|---|
| `pnpm dev` | Watch-build the lib and run the demo site |
| `pnpm build` | Build the lib, then the demo site |
| `pnpm test` | Run jest tests in `packages/react-chess` |
| `pnpm lint` | Lint all workspaces |

Package-scoped (from `packages/react-chess/`):

| Command | What it does |
|---|---|
| `npm run build` | One-shot rollup build |
| `npm run dev` | Rollup watch mode |
| `npm test` | Jest |
| `npm run test:watch` | Jest watch mode |
| `npm run lint` | ESLint |

## How the build works

`packages/react-chess/rollup.config.js` produces a multi-output build:

- `dist/index.js` — CJS bundle
- `dist/index.esm.js` — ESM bundle
- `dist/index.d.ts` and per-file `.d.ts` — type declarations from `@rollup/plugin-typescript`
- `dist/assets/index.css` — extracted by `rollup-plugin-postcss`, then concatenated with chessground's base/theme/piece stylesheets by the inline `concatChessgroundCSS` plugin
- `dist/styles.d.ts` — empty `export {};` stub so the `./styles` subpath import is recognised by TS under `node16`/`nodenext`/`bundler` module resolution

Peer/runtime deps (`react`, `react-dom`, `chessground`, `chess.js`, `lucide-react`, `clsx`) are externalised so consumers can dedupe them.

## Package exports

The library is exported via the `exports` map in `packages/react-chess/package.json`:

```jsonc
{
  "exports": {
    ".": {
      "import": "./dist/index.esm.js",
      "require": "./dist/index.js",
      "types": "./dist/index.d.ts"
    },
    "./styles": {
      "types": "./dist/styles.d.ts",
      "default": "./dist/assets/index.css"
    }
  }
}
```

If you add a new public subpath (e.g. `./icons`), add it here too — and emit a `.d.ts` stub if it's a non-TS asset, otherwise consumers using strict module resolution will see TS2307.

## Where things live

| What | Where |
|---|---|
| Top-level convenience wrapper | `src/Chessboard.tsx` |
| Shared state / context | `src/context/ChessProvider.tsx` |

| Tests | `src/__tests__/` |

## Testing

```sh
pnpm test
```

Tests run with jest + jest-environment-jsdom + ts-jest. New components should ship with a corresponding test in `src/__tests__/`.

## Release flow

Releases are manual. Steps:

1. **Bump the version** in `packages/react-chess/package.json` (semver: patch for fixes, minor for additive features, major for breaking changes).
2. **Build** to refresh `dist/`:
   ```sh
   cd packages/react-chess
   npm run build
   ```
3. **Commit** the bump + any source changes on `master`:
   ```sh
   git add -A
   git commit -m "fix: <what changed>"
   git push origin master
   ```
   The repo follows conventional commit prefixes: `fix:`, `feat:`, `chore:`, `docs:`.
4. **Publish from the package directory** (this is important — `npm publish` from the monorepo root will fail with `Cannot read properties of null (reading 'prerelease')` because the root `package.json` has no `version` field):
   ```sh
   cd packages/react-chess
   npm whoami        # if this errors, run `npm login` first
   npm publish --access public
   ```
   The `--access public` flag is required because `@mdwebb/react-chess` is a scoped package and would default to private otherwise.
5. **Approve the 2FA prompt.** npm uses a web-based 2FA flow — it prints a `https://www.npmjs.com/auth/cli/...` URL. Open it in your browser and approve. No authenticator app is required.
6. **Verify** the registry shows the new version:
   ```sh
   npm view @mdwebb/react-chess version
   ```

### Release troubleshooting

- **`Cannot read properties of null (reading 'prerelease')`** — you ran `npm publish` from the monorepo root instead of `packages/react-chess`. `cd` into the package and retry.
- **`Cannot read properties of null (reading 'matches')`** during `npm install` in a consumer project — known arborist bug in older npm. Upgrade with `npm i -g npm@latest`, then clear the consumer's `node_modules` and `package-lock.json` and reinstall.
- **`EOTP` with a redacted `***` URL** — that redaction is from terminal-capture tooling, not npm. Run `npm publish` directly in your shell and the real URL will print.

## Filing issues / PRs

- Open issues at https://github.com/matt-d-webb/react-chess/issues with a minimal reproduction.
- For PRs, branch from `master`, keep changes focused, and include a test if you're touching component behavior.
